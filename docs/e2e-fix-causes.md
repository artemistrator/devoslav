# Причины сбоев E2E-сценария (идея → планы → задачи → Auto Execution)

Документ составлен по результату прохода сценария в браузере (порт 3002): ввод идеи «создай hello world», генерация планов, выбор плана, задачи, нажатие Auto Execution и просмотр консоли Execution.

---

## 1. Генерация планов по новой идее не срабатывает

**Что сделано:** На главной введена идея «создай hello world», нажата «Отправить». Кнопка перешла в «Генерируем…», затем снова в «Отправить». В блоке «Планировщик» по-прежнему текст «Здесь появятся варианты после отправки идеи» — планы не отобразились.

**Ранее наблюдалось:** Уведомление «Не удалось получить планы. Ответ модели не удалось разобрать (ожидался JSON с тремя планами).»

**Вероятные причины:**

- **Парсинг ответа LLM:** API `decompose-idea` ожидает строгий JSON с полем `plans` (массив из трёх планов). Ответ модели может содержать пояснения до/после JSON, обёртку в markdown (например, ```json ... ```) или невалидный JSON — парсер падает, фронт получает ошибку и не показывает планы.
- **Разные провайдеры/модели:** При смене провайдера (OpenAI/Anthropic/Z.ai) или модели формат ответа может отличаться; парсер не учитывает все варианты.

**Рекомендации по исправлению:**

- В `app/api/decompose-idea/route.ts` (или там, где парсится ответ):
  - Вынести JSON из ответа: обрезать markdown-блоки (```json ... ```), искать первую подходящую подстроку `{ ... }` и парсить только её.
  - Использовать устойчивый парсер (например, поиск границ `{`/`}` и `JSON.parse` по подстроке) и при ошибке парсинга логировать сырой ответ и возвращать клиенту понятное сообщение (например, «Ответ модели не удался разобрать» с опциональным `raw` в ответе для отладки).
- На фронте при ошибке от API показывать уведомление с текстом ошибки (уже есть) и не сбрасывать ввод идеи, чтобы пользователь мог повторить или скорректировать запрос.

## 2. Воркер выполняет только задачи в статусе TODO

**Что сделано:** Открыт существующий проект «сделай hello world html страницу», выбран план «Вариант 1: Прямой путь», открыты задачи. Одна задача уже в статусе **IN PROGRESS**. Нажата «Start Auto Execution» → модалка → «Start Execution». Появилось уведомление «Execution started», вкладка Execution показывает «Running», но в консоли (TERMINAL) только два сообщения: «Ready to execute commands. Type a command or message to get started.» Новых логов от агента (генерация промпта, task_started, шаги плана, отчёт) не появилось.

**Причина (по коду):** В `app/api/execution-sessions/run/route.ts` воркер загружает задачи так:

```ts
const tasks = await prisma.task.findMany({
  where: {
    planId: session.planId || undefined,
    status: "TODO",  // <-- только TODO
  },
  ...
});
```

Если все задачи плана уже в статусе IN_PROGRESS (или REVIEW/DONE), воркер получает **0 задач**, не вызывает ни `generateTaskPrompt`, ни `executeTask`, поэтому в ExecutionLog ничего не пишется и в консоли пользователь не видит активности.

**Рекомендации по исправлению:**

- **Вариант A (минимальный):** При старте Auto Execution переводить «зависшие» задачи из IN_PROGRESS обратно в TODO, если с момента последнего обновления прошло больше N минут (или если по задаче нет комментария/отчёта). Тогда следующий запуск воркера подхватит их.
- **Вариант B (расширенный):** Воркеру разрешить обрабатывать не только TODO, но и IN_PROGRESS (например, `status: { in: ["TODO", "IN_PROGRESS"] }`) с ограничением: только задачи без недавнего комментария от агента или с флагом «нужен повтор». Так можно дождать выполнения «застрявших» задач без ручного сброса статуса.
- **Вариант C (явное поведение в UI):** На странице плана перед «Start Auto Execution» показывать предупреждение, если нет задач в TODO: «Нет задач в статусе To Do. Переведите задачи в To Do или создайте новые.» И/или кнопка «Сбросить задачи в To Do» для выбранного плана (осторожно с уже выполненными).

Выбор варианта — по продукту; минимально достаточно A или явного C, чтобы пользователь понимал, почему после нажатия Auto Execution «ничего не происходит».

## 3. В консоли Execution нет логов воркера

**Что видно:** Во вкладке Execution в блоке TERMINAL отображаются только системные сообщения «Ready to execute commands...». Сообщений от ExecutionAgent (промпт, шаги, writeFile, отчёт) нет.

**Возможные причины:**

- **Логи не создаются:** Из-за п.2 воркер не обрабатывает ни одной задачи → событий в `ExecutionLog` нет → поллинг возвращает пустой массив, в консоль нечего добавлять.
- **Поллинг логов падает:** Раньше наблюдалась ошибка «Failed to fetch» при запросе к `/api/execution-sessions/:sessionId/logs`. Если запрос падает (сеть, 500 от API), новые логи не подтягиваются и в UI не отображаются. Уже сделано: возврат 500 с JSON при ошибке в API логов и обработка ошибки в `ExecutionConsole` с сообщением пользователю — стоит убедиться, что в среде выполнения нет CORS/сетевых ошибок и что `prisma.executionLog` доступен (после `prisma generate`).

**Рекомендации:**

- Убедиться, что при реальном выполнении хотя бы одной задачи (TODO → воркер подхватил) в БД появляются записи в `ExecutionLog` и что запрос `GET /api/execution-sessions/:id/logs` возвращает 200 и массив логов. При необходимости добавить краткий лог на сервере при записи в `ExecutionLog`, чтобы по логам сервера проверить, что события доходят до БД.
- Если «Failed to fetch» повторяется: проверить переменную окружения для внутренних вызовов (например, `INTERNAL_APP_URL` при вызове воркера из `start`) и что приложение доступно по тому же origin/порту, с которого фронт ходит за логами.

## 4. Files: «Loading file tree...» не завершается

**Что видно:** В панели Files на вкладке Execution отображается «Loading file tree...» и состояние не меняется.

**Вероятные причины:**

- Запрос к API файлов (например, `/api/files?projectId=...`) зависает, возвращает ошибку или пустой/неожиданный формат; компонент не переходит в состояние «успех» или «ошибка».
- Для «cloud»-режима дерева файлов проекта на сервере может не быть (нет синхронизированной папки) — тогда API может корректно возвращать пустое дерево, но фронт должен обрабатывать это и показывать «No files» вместо бесконечной загрузки.

**Рекомендации:**

- В компоненте, который запрашивает дерево файлов (например, в ExecutionConsole или дочернем блоке Files), добавить таймаут и обработку ошибки: по истечении таймаута или при `!res.ok` выставлять состояние ошибки и показывать текст вида «Не удалось загрузить дерево файлов» вместо бесконечного «Loading file tree...».
- На бэкенде для `GET /api/files` при отсутствии файлов возвращать 200 и `{ root: null }` или `{ root: { children: [] } }`, чтобы фронт мог отрисовать «пустое дерево» и убрать индикатор загрузки.

## 5. Краткий чеклист по сценарию

| Шаг                         | Ожидание                         | Наблюдение / проблема                    |
|----------------------------|-----------------------------------|------------------------------------------|
| Ввод идеи «создай hello world» | Текст в поле, кнопка «Отправить» | Ок                                       |
| Генерация планов           | Три варианта плана в блоке «Планировщик» | Планы не появляются, ошибка парсинга JSON |
| Выбор плана, открытие задач| Переход на страницу плана с задачами     | Ок (на существующем проекте)             |
| Start Auto Execution       | Модалка → Start Execution → сессия запущена | Ок, уведомление «Execution started»       |
| Консоль Execution          | Логи агента (промпт, шаги, отчёт) | Только «Ready to execute commands»        |
| Причина пустой консоли     | Воркер обрабатывает задачи       | Воркер берёт только TODO; задача была IN_PROGRESS → 0 задач |

Исправление п.1 (парсинг планов) и п.2 (обработка TODO/IN_PROGRESS и/или явное поведение в UI) устранит основные разрывы сценария и даст видимые логи в консоли при реальном запуске задач.
