## Fixnew-1802 — Система тикетов (DB + UI + Worker + ExecutionAgent)

Дата: 18.02.2026  
Автор: Cursor AI (Execution / Backend / Frontend)

В этом файле зафиксированы изменения по трём связанным задачам:

1. Введение сущности **Ticket** в БД + API + UI (создание тикетов из задач).
2. Обработка тикетов в **воркере** (`runExecutionSession`): приоритетная очередь тикетов.
3. Реализация **`executeTicket`** в `ExecutionAgent`: повторное выполнение исходной задачи с учётом новых требований.

Ниже разбор по шагам с объяснением того, что и зачем было сделано.

---

## 1. Модель Ticket в Prisma + API + UI в PlanPageClient

### 1.1. Изменения в `prisma/schema.prisma`

**Цель:** Добавить сущность лёгкого тикета, связанного с проектом и (опционально) с исходной задачей.

Сделано:

- В модель `Project` добавлено поле:

- `tickets Ticket[]` — связь "проект → множество тикетов".

- Добавлен новый enum:

- `TicketStatus { OPEN, IN_PROGRESS, DONE, REJECTED }`.

- Добавлена модель `Ticket`:

  - `id String @id @default(cuid())` — идентификатор тикета.
  - `projectId String` + связь `project Project @relation(..., onDelete: Cascade)` — каждый тикет принадлежит проекту, при удалении проекта тикеты каскадно удаляются.
  - `title String` — короткий заголовок тикета.
  - `description String @db.Text` — развёрнутое описание проблемы/улучшения.
  - `status TicketStatus @default(OPEN)` — статус тикета.
  - `relatedTaskId String?` — опциональный ID исходной задачи, которую этот тикет чинит/уточняет.
  - `createdAt DateTime @default(now())`, `updatedAt DateTime @updatedAt` — стандартные временные метки.

Таким образом, у нас появилось хранилище для лёгких запросов/правок, связанных с существующими задачами плана.

### 1.2. API для создания тикета — `POST /api/tickets`

Файл: `app/api/tickets/route.ts`.

**Цель:** дать фронтенду простой endpoint для создания тикетов.

Сделано:

- Реализован `POST`‑хендлер, который:

  - Читает тело запроса и безопасно извлекает:
    - `projectId: string`
    - `title: string`
    - `description: string`
    - `relatedTaskId?: string`
  - Валидирует вход:
    - `projectId`, `title`, `description` обязательны и не могут быть пустыми после `trim`.
  - Создаёт запись `Ticket` через Prisma:
    - `status` всегда `"OPEN"` при создании (воркер уже потом переводит в `IN_PROGRESS`, `DONE`, `REJECTED`).
  - Возвращает JSON `{ success: true, ticket }` со статусом `201`, либо осмысленные ошибки на `400/500`.

Таким образом, API для тикетов максимально простой: один endpoint, один сценарий — создание открытого тикета.

### 1.3. UI: кнопка Ticket и модалка "Create a Ticket" в PlanPageClient

**Цель:** дать пользователю возможность из завершённой/проваленной задачи создать тикет с описанием проблемы или доработки.

Сделано в двух компонентах:

#### a) `TaskListClient.tsx` — кнопка Ticket на задачах DONE/REVIEW

- В таблице задач (режим `kanban`‑лист в `PlanPageClient`) для каждой строки, соответствующей задаче:

  - Добавлен новый проп `onCreateTicket?: (task: TaskDetail) => void`.
  - Для задач со статусами `DONE` или `REVIEW` рендерится кнопка **Ticket** (иконка `Bug` из `lucide-react`).
  - При клике:
    - Останавливается всплытие (`e.stopPropagation()`), чтобы не триггерить открытую карточку.
    - Вызывается `onCreateTicket(task)` — делегирование логики на родителя (`PlanPageClient`).

Таким образом, любая завершённая или отправленная на ревью задача получает лёгкий вход в создание тикета.

#### b) `PlanPageClient.tsx` — модалка создания тикета

- В `PlanPageClient` добавлены состояния:

  - `ticketModalTask: TaskDetail | null` — текущая задача, для которой создаётся тикет.
  - `ticketTitle: string` — заголовок тикета (по умолчанию копирует заголовок задачи, но можно изменить).
  - `ticketDescription: string` — описание проблемы/запроса.
  - `ticketSubmitting: boolean` — флаг отправки формы (для дизейбла и защиты от дабл‑кликов).

- Вызов `TaskListClient` расширен:

  - Передаётся `onCreateTicket`, который:
    - Ставит `ticketModalTask = task`.
    - Заполняет `ticketTitle = task.title`.
    - Очищает `ticketDescription`.

- Добавлен компонент `Dialog` (модалка) с формой:

  - Заголовок: `Ticket for task: “<task.title>”`.
  - Поля:
    - `Title` — `Input` с `ticketTitle`.
    - `Description` — `<textarea>` с `ticketDescription`.
  - Кнопки:
    - `Cancel` — закрывает модалку и очищает состояние.
    - `Create Ticket` — сабмит форм, отключается, если поля пустые или идёт отправка.

- При сабмите:

  - Отправляется `POST` на `/api/tickets` с телом:
    - `projectId` (из пропсов PlanPageClient),
    - `title` (trim),
    - `description` (trim),
    - `relatedTaskId` (ID исходной задачи).
  - При успехе модалка закрывается, локальное состояние очищается.
  - Ошибки логируются в консоль, но не ломают приложение.

Результат: пользователь, находясь в UI плана, может для любой завершённой или проваленной задачи нажать Ticket, описать проблему/улучшение и сохранить тикет в БД.

## 2. Обработка тикетов в воркере (`runExecutionSession`)

Файл: `app/api/execution-sessions/run/route.ts`.

**Цель:** сделать воркер двуочередным:

- Сначала обрабатывать **тикеты** со статусом `OPEN`.
- Если тикетов нет — работать по старой схеме с задачами `TODO`.

### 2.1. Поиск открытых тикетов

После загрузки `ExecutionSession` и возможного ресета задач `IN_PROGRESS → TODO` для плана:

- Добавлен запрос:

- `prisma.ticket.findMany({ where: { projectId: session.projectId, status: "OPEN" }, orderBy: { createdAt: "asc" } })`.

Это формирует приоритетную очередь тикетов для текущего проекта.

### 2.2. Обработка тикета (если он есть)

Если `openTickets.length > 0`:

1. Берётся один тикет — самый старый по `createdAt`.
2. Пытаемся пометить его как `IN_PROGRESS`:

   - Если апдейт неудачен — логируем, но стараемся продолжить (защита от гонок и непоследовательных состояний).

3. Вызывается **новый метод** агента:

   - `const result = await agent.executeTicket(currentTicket);`

4. По результату:

   - Если `result.success === true` — тикет обновляется в статус `DONE`.
   - Иначе — тикет обновляется в статус `REJECTED`.
   - Любые крэши вокруг выполнения тикета дополнительно перехватываются и также приводят к `REJECTED`, чтобы тикет не завис в `IN_PROGRESS`.

Важно: в этом режиме **обычные задачи не берутся**, сессия выполняет только один тикет за прогон. Логика обработки задач `TODO` исполняется только если тикетов нет.

### 2.3. Обработка задач, если тикетов нет

Если `openTickets.length === 0`:

- Воркер возвращается к прежней логике:

  - Находит все задачи с `status: "TODO"` данного плана.
  - В цикле по задачам:
    - Генерирует промпт через `generateTaskPrompt(task.id, false, false)`.
    - Вызывает `agent.executeTask(task.id, generatedPrompt)`.
    - Дальше работает существующий QA‑цикл:
      - Сохранение отчёта в комментариях.
      - `verifyTaskCompletion`, автодополнение evidence при REJECT и повторные попытки.
      - Обновление статуса задач, логов и, при необходимости, статуса проекта.

Таким образом, очередь теперь выглядит так:

1. Если есть тикеты `OPEN` для проекта — берём один тикет и обрабатываем его.
2. Если тикетов нет — обрабатываем обычные задачи плана в статусе `TODO`.

## 3. Реализация `executeTicket` в `ExecutionAgent`

Файл: `lib/agents/execution-agent.ts`.

**Цель:** реализовать метод `executeTicket(ticket)`, который:

1. Берёт исходную задачу (`originalTask`) по `ticket.relatedTaskId`.
2. Получает актуальное состояние проекта.
3. Генерирует новый промпт задачи, встраивая в него требования тикета.
4. Запускает внутреннюю логику `executeTaskInner` для **исходной задачи**, но уже с обновлённым промптом.

Это позволяет не придумывать задачу заново, а "перерисовывать чертёж" с учётом новых требований.

### 3.1. Подготовка: импорты и публичный метод

Добавлены импорты:

- `generateProjectContext` — для полного контекста проекта.
- `generateTaskPrompt` — для генерации/перегенерации промпта задачи с учётом новых требований.

Также добавлен публичный метод:

- `async executeTicket(ticket: Ticket): Promise<ExecuteTaskResult>`:

  - Управляет флагами `isRunning` / `isPaused`.
  - Делегирует основную работу в приватный `executeTicketInner`.
  - В случае крэша логирует ошибку, пишет event с `ticketId` в `executionLog` и возвращает `{ success: false }`.

Это обертка, совместимая по семантике с `executeTask`.

### 3.2. executeTicketInner: пошаговая логика

Приватный метод `executeTicketInner(ticket: Ticket)` реализует требуемые шаги:

#### (a) Загрузка оригинальной задачи

- Если `ticket.relatedTaskId` отсутствует:

  - Логируется ошибка `Ticket ... has no relatedTaskId`.
  - Метод возвращает `{ success: false }`, т.к. нет задачи, к которой можно применить тикет.

- Иначе:

  - Через `prisma.task.findUnique` загружается `originalTask` c:
    - `plan` и `plan.project` — для доступа к ID проекта.
  - Если задача не найдена или у неё нет проекта — логируется ошибка и возвращается `{ success: false }`.

Таким образом, мы всегда опираемся на реальную исходную задачу, а не на абстрактный текст тикета.

#### (b) Актуальный контекст проекта

- Берётся `projectId = originalTask.plan.project.id`.
- Вызывается:

  - `generateProjectContext(projectId, { includeDetails: true, maxTasks: 20 })`.

- Результат — строка с:

  - Идеей проекта, активным планом, списком выполненных задач, текущими блокерами, предстоящими задачами и ADR‑файлами.

- Любые ошибки при генерации контекста не фатальны:

  - Логируются как ошибка, но выполнение продолжается даже с пустым `projectContext`.

Это даёт "снимок" актуального состояния, на который может опираться новый промпт.

#### (c) Формирование дополнительного требования из тикета

- Формируется строка `extraRequirement`, которая содержит:

  - Фразу в духе:

    - `Regenerate instructions for this task, but with this new requirement from ticket "<ticket.title>": "<ticket.description>"`.

  - При наличии `projectContext` — он добавляется в конец, как "Use the following up-to-date project state...".

Идея: дать генератору промпта явную инструкцию "перегенерировать промпт с учётом этого тикета" и одновременно подсунуть свежий контекст проекта.

#### (d) Новый `generatedPrompt` через generateTaskPrompt

- Вызывается:

  - `generateTaskPrompt(originalTask.id, true, false, extraRequirement)`.

  - `forceRegenerate = true` — насильно регенерируем промпт, даже если ранее был сохранён.
  - `skipSave = false` — новый промпт сохраняется в `task.generatedPrompt`, чтобы UI и другие части системы видели актуальный текст.
  - `extraRequirement` — как описано выше, встраивается в содержимое промпта.

- При любой ошибке:

  - Логируется подробное сообщение с ID тикета и задачи.
  - Метод возвращает `{ success: false }`.

Это соответствует идее: "перерисовать чертёж задачи" с учётом тикета.

#### (e) Исполнение исходной задачи с новым промптом

- После успешной генерации:

  - Логируется сообщение вида:

    - `Executing ticket "<ticket.title>" by re-running original task <taskId> with updated prompt`.

  - Вызывается:

    - `return this.executeTaskInner(originalTask.id, regeneratedPrompt);`

- Таким образом:

  - Используется вся существующая логика выполнения задачи:
    - Генерация execution‑плана (LLM → JSON со step’ами).
    - Выполнение tool‑шагов (`executeCommand`, `readFile`, `writeFile`, `searchKnowledge`, `webSearch`).
    - Верификация через `verificationCriteria` (автоматические команды, артефакты).
    - Генерация развернутого DEVOPS‑отчёта и прогон через QA.
    - Возможный билд (если включён `runBuildAfterTask`).
  - Но инструкции для задачи уже включают требования тикета и актуальный project state.

Результат: тикет не порождает "левую" задачу, а аккуратно переиспользует существующую, добавляя новые требования в её промпт. Это делает поведение детерминированным и предсказуемым: все доработки агрегируются вокруг исходной задачи, а не размножают её.

## Итоговое поведение всей связки

1. **Пользователь** из UI (`PlanPageClient`) создаёт тикет для завершённой/проваленной задачи (DONE/REVIEW).
2. **API `/api/tickets`** сохраняет тикет со статусом `OPEN`, привязанным к проекту и исходной задаче.
3. **Воркер** при запуске сессии:
   - Сначала ищет `Ticket` со статусом `OPEN`.
   - Если нашёл — берёт один, ставит `IN_PROGRESS` и вызывает `agent.executeTicket(ticket)`.
   - Если тикетов нет — работает по старой схеме с задачами `TODO`.
4. **`ExecutionAgent.executeTicket`**:
   - Находит исходную задачу.
   - Генерирует подробный контекст проекта.
   - Регенерирует промпт задачи с дополнительным блоком требований из тикета.
   - Запускает `executeTaskInner` для исходной задачи с новым промптом и полным QA‑циклом.
5. **Статус тикета** по результату:
   - `DONE`, если выполнение прошло успешно.
   - `REJECTED`, если произошёл крэш или агент не смог успешно завершить задачу с учётом тикета.

Эта архитектура позволяет:

- Разделить **вход** (тикеты) и **исполнение** (переиспользование исходных задач).
- Сохранить единый источник правды на уровне Task’ов (Task остаётся основной единицей работы).
- Дать пользователю удобный UX для "переоткрытия" или уточнения задач без ручного редактирования плана.
