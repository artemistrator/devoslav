🚀 AI Orchestrator - Product Roadmap
Этот документ описывает стратегические направления развития продукта, основанные на нашем мозговом штурме. Цель — превратить текущий MVP в полноценную, "живую" и масштабируемую AI-платформу для разработки.

🎯 Эпик 1: "Живой Проект" (Stateful Evolution)
Превращаем одноразовые проекты в долгосрочные, развивающиеся продукты.

1.1. Режим "Эволюции" (Доработка существующих проектов)
Проблема: Сейчас каждый проект создается с нуля. Невозможно доработать уже существующий.

Решение:

На главной странице добавить переключатель: "New Project" / "Evolve Existing Project".

При выборе "Evolve" показывать Dropdown со списком существующих проектов.

Backend: API /decompose-idea должен принимать baseProjectId.

AI Logic: Архитектор получает специальный промпт "You are MODIFYING an existing project..." и полный доступ к кодовой базе baseProjectId для анализа.

Ожидаемый результат: Пользователь может выбрать старый проект, написать "Добавь блог" и получить только те задачи, которые нужны для реализации блога, без пересоздания всего приложения.

1.2. Сборка и Экспорт
Проблема: Результат работы AI (код) живет только внутри Docker.

Решение:

Добавить в ExecutionAgent финальный шаг npm run build (для веб-проектов).

Создать API, который архивирует папку с билдом (dist, .next) или весь проект.

В UI добавить кнопку "Download Build" или "Export to ZIP".

Ожидаемый результат: Пользователь может в любой момент скачать готовый к деплою артефакт.

🎯 Эпик 2: "Контроль и Управление" (Human-in-the-Loop)
Даем пользователю "руль" для управления AI, а не просто "педаль газа".

2.1. Ручное добавление задач и "Heavy Replan"
Проблема: Пользователь не может влиять на уже сгенерированный план.

Решение:

UI: Добавить кнопку "Add Manual Task" на странице плана.

UX: Перед добавлением показывать AlertDialog: "This will trigger a full plan re-evaluation. Proceed?".

Backend: При создании задачи пользователем, запускать "тяжелый" реплан (ArchitectAgent.fullReplan).

AI Logic: Агент-архитектор получает все задачи (старые + новую) и промпт: "Analyze the entire plan for conflicts and dependencies. Generate a NEW, OPTIMIZED plan."

Ожидаемый результат: План разработки гибко адаптируется под новые вводные от пользователя.

2.2. Интерактивный Replan (Diff View)
Проблема: "Heavy Replan" может быть слишком "черным ящиком".

Решение:

Backend: fullReplan должен возвращать не просто новый план, а "diff" — что было удалено, что добавлено, что изменено.

UI: Показать этот diff пользователю в наглядном виде:

- [УДАЛЕНО] (красный)

+ [ДОБАВЛЕНО] (зеленый)

~ [ИЗМЕНЕНО] (желтый)

Добавить кнопки "Approve Plan Changes" / "Cancel".

Ожидаемый результат: Пользователь получает полный контроль над изменениями плана.

🎯 Эпик 3: "Самообучение" (Live Insights & Simulation)
Учим систему делать выводы из своего же опыта и предсказывать проблемы.

3.1. "Живые" Инсайты (Агент-Рефлексолог)
Проблема: Инсайты генерируются только после удаления проекта.

Решение:

Запускать "Агента-Рефлексолога" после завершения крупных этапов (например, батча из 5 задач).

Источники данных для анализа:

Комментарии QA: Повторяющиеся замечания.

Ошибки консоли: Частые ошибки npm, build.

Выбор инструментов: Анализ того, какие инструменты агент использует чаще.

UI: Предлагать пользователю 3-5 инсайтов с кнопкой "Save to Global Knowledge".

Ожидаемый результат: База знаний (GlobalInsight) пополняется полезными уроками из каждого проекта, а не только из удаленных.

3.2. Симуляция перед выполнением (Dry Run)
Проблема: Агент может сгенерировать логичный, но технически невыполнимый план.

Решение:

Перед ExecutionAgent запускать "Агента-Симулятора".

AI Logic: Симулятор получает steps и промпт: "Analyze this plan for potential errors. Will npm install X work in a Python project?".

Он возвращает массив warnings: [].

Workflow:

Если warnings есть, показать их пользователю и отправить Архитектору на доработку.

Если warnings нет, запускать ExecutionAgent.

Ожидаемый результат: Сокращение количества глупых ошибок и сэкономленные токены/время.
