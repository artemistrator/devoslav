# AI Orchestrator — Руководство пользователя

AI Orchestrator — это интеллектуальная система управления проектами и автоматизации разработки, которая превращает идеи в структурированные планы развития и автоматически реализует их. Система использует множество AI-агентов, работающих параллельно, и включает продвинутые механизмы анализа кода, оценки сложности и обучения на ошибках.

## 1. Декомпозиция идей

Система принимает описание идеи на естественном языке и автоматически генерирует несколько архитектурных планов развития проекта. Пользователь описывает свою идею, а AI Orchestrator анализирует её и предлагает различные варианты реализации с учётом различных технологических стеков.

### Основные возможности

- **Множественные планы**: Для каждой идеи генерируется несколько архитектурных планов с разными подходами к реализации — от консервативного до инновационного
- **Технологические стеки**: Каждый план включает рекомендуемый набор технологий с обоснованием выбора
- **Анализ требований**: AI автоматически извлекает функциональные и нефункциональные требования из описания идеи
- **Оценка сложности**: Система присваивает каждому плану уровень сложности (S, M, L, XL), который влияет на количество генерируемых задач

### Входные данные

Для декомпозиции пользователь предоставляет:

- Текстовое описание идеи — что нужно построить и какую проблему решить
- Дополнительный контекст — существующие системы, ограничения, требования к производительности
- При желании — базовый проект для эволюции существующей системы

## 2. Управление планами

Каждый проект может содержать несколько планов. Планы представляют собой различные варианты архитектурного решения одной и той же задачи.

### Структура плана

План содержит детальную информацию о выбранном подходе к реализации:

- **Описание архитектуры**: Общая структура системы и её компоненты
- **Технологический стек**: Перечень технологий, библиотек и фреймворков
- **Список задач**: Декомпозиция плана на конкретные задачи
- **Оценка сложности**: Уровень сложности (S/M/L/XL) с обоснованием
- **Критерии приёмки**: Условия, по которым план считается успешно реализованным

### Выбор плана

Пользователь может просмотреть все сгенерированные планы, изучить их отличия и выбрать наиболее подходящий для дальнейшей реализации. После выбора плана система генерирует задачи для его выполнения.

## 3. Система задач

Задачи — это атомарные единицы работы, на которые декомпозируется выбранный план. Каждая задача содержит всю необходимую информацию для её выполнения.

### Жизненный цикл задачи

Задачи проходят через следующие статусы:

- **TODO**: Задача создана и ожидает выполнения
- **IN_PROGRESS**: Задача взята в работу
- **REVIEW**: Работа над задачей завершена и находится на проверке
- **WAITING_APPROVAL**: Ожидает ручного подтверждения от пользователя
- **DONE**: Задача завершена и подтверждена

### Структура задачи

Каждая задача содержит:

- **Название и описание**: Что нужно сделать
- **Вердикт (Prompt)**: Детальная инструкция для AI-агента с контекстом и требованиями
- **Критерии верификации**: Конкретные условия, по которым QA агент проверит выполнение
- **Исполнитель**: Тип агента, который должен выполнить задачу (Frontend, Backend, DevOps, Teamlead, Cursor, QA)
- **Зависимости**: Ссылки на задачи, которые должны быть выполнены перед этой
- **Вложения**: Файлы, изображения, скриншоты, относящиеся к задаче

### Зависимости между задачами

Система поддерживает граф зависимостей между задачами. Это позволяет:

- Определять порядок выполнения задач
- Автоматически перепланировать при изменении графа
- Визуализировать связи между задачами
- Анализировать влияние изменений

## 4. AHP — протокол параллельных агентов

AHP (Agent Hive Protocol) — это современный движок выполнения задач, использующий параллельную работу множества AI-агентов. В отличие от последовательного подхода, AHP позволяет нескольким агентам работать одновременно, обмениваясь сообщениями через общую шину.

### Архитектура AHP

Система состоит из нескольких типов агентов, каждый из которых отвечает за свою область:

- **Frontend-агент**: Работает с пользовательским интерфейсом, стилями, компонентами React/Vue
- **Backend-агент**: Реализует серверную логику, API, базы данных
- **DevOps-агент**: Настраивает инфраструктуру, CI/CD, контейнеризацию
- **Teamlead-агент**: Координирует работу, создаёт документацию, проводит ревью
- **Cursor-агент**: Выполняет работу в IDE, пишет и редактирует код
- **QA-агент**: Проверяет качество выполненной работы

### Меж-Agent коммуникация

Агенты общаются через систему сообщений, которая обеспечивает:

- **Асинхронный обмен**: Агенты не блокируют друг друга
- **Корреляцию сообщений**: Связанные сообщения группируются по correlation ID
- **Ответные цепочки**: Сообщения могут ссылаться друг на друга (replyTo)
- **Отслеживание статуса**: Каждое сообщение имеет статус (PENDING, PROCESSING, PROCESSED, FAILED)

### Типы сообщений

- **TASK_REQUEST / TASK_RESPONSE**: Запросы на выполнение задач
- **STYLE_REQUEST / STYLE_RESPONSE**: Согласование стилей кода между агентами
- **QA_REQUEST / QA_RESPONSE**: Запросы верификации
- **ARCHITECT_REQUEST / ARCHITECT_RESPONSE**: Запросы к архитектору

### Выбор движка

При запуске автоматического выполнения пользователь может выбрать движок:

- **Legacy**: Классический последовательный подход (по умолчанию)
- **AHP**: Параллельный подход с множеством агентов

## 5. Оценка сложности проекта

Система автоматически анализирует идею и оценивает сложность будущего проекта. Это помогает реалистично планировать ресурсы и количество необходимых задач.

### Уровни сложности

- **S (Small)**: Простые проекты — лендинги, небольшие CRUD-приложения, до 10 задач
- **M (Medium)**: Средние проекты — полноценные веб-приложения с аутентификацией, 20-40 задач
- **L (Large)**: Сложные проекты — микросервисная архитектура, интеграции, 50-100 задач
- **XL (Extra Large)**: Очень сложные проекты — распределённые системы, ML-компоненты, 100+ задач

### Факторы оценки

При определении сложности AI учитывает:

- Количество функциональных требований
- Необходимость интеграций с внешними системами
- Требования к безопасности и масштабируемости
- Наличие нестандартных технических решений
- Необходимость миграции данных

## 6. Эволюция проекта

Функция эволюции позволяет расширять существующие проекты, а не создавать новые с нуля. Это полезно для доработки функционала, исправления багов или добавления новых возможностей.

### Как работает эволюция

При указании существующего проекта в качестве базового:

- Система загружает контекст завершённых задач
- Анализирует используемый технологический стек
- Изучает существующую архитектуру
- Генерирует план, совместимый с текущей кодовой базой
- Копирует релевантные файлы в новый проект

### Применение

Эволюция используется когда нужно:

- Добавить новый функционал в работающий проект
- Провести рефакторинг существующего кода
- Обновить технологический стек
- Исправить архитектурные проблемы

## 7. Автоматическое выполнение

Автоматическое выполнение (Auto Execution) позволяет системе самостоятельно реализовывать план без участия человека. Execution Agent берёт задачи из очереди, генерирует детальные промпты, выполняет команды и верифицирует результаты.

### Режимы выполнения

- **Local (локальный)**: Команды выполняются на машине клиента через sync-client
- **Cloud (облачный)**: Команды выполняются на удалённых серверах

### Управление выполнением

- **Запуск**: Начать автоматическое выполнение выбранного плана
- **Пауза**: Приостановить выполнение для ручного вмешательства
- **Возобновление**: Продолжить с места остановки
- **Остановка**: Полностью завершить сессию выполнения

### Контроль качества

- **Лимит затрат**: Максимальная сумма, которую можно потратить за сессию
- **Auto-approve**: Автоматическое подтверждение команд без участия человека
- **Ручное подтверждение**: Каждая команда требует одобрения перед выполнением
- **Консоль мониторинга**: В реальном времени отображается ход выполнения, вывод команд и ответы AI

### Логирование

Каждая сессия выполнения записывает:

- Все отправленные команды
- Вывод команд (stdout/stderr)
- ответы AI-агентов
- Ошибки и их описание
- Время выполнения каждого шага
- Потраченные токены и стоимость

## 8. QA-верификация и рефлексолог

После выполнения каждой задачи QA-агент проверяет результат работы, анализирует ошибки и принимает решение о качестве выполнения.

### QA-агент

QA-агент отвечает за верификацию задач и обеспечивает качество кода. Его возможности:

- **Чтение файлов**: Агент может читать файлы проекта для проверки реализации
- **Веб-поиск**: Интеграция с Tavily API для поиска актуальной документации
- **Строгая верификация**: Задача принимается только при наличии конкретных доказательств выполнения — логов тестов, успешной сборки, вывода команд
- **Детальный анализ ошибок**: При отказе агент предоставляет конкретную обратную связь с указанием что именно пошло не так

### Критерии верификации

Каждая задача содержит явные критерии приёмки, которые QA-агент проверяет:

- Успешная сборка проекта
- Прохождение тестов
- Корректная работа API-эндпоинтов
- Соответствие кода стилю проекта
- Отсутствие критических ошибок в логах

### Рефлексолог

Рефлексолог — это AI-агент, который запускается после завершения сессии выполнения. Он анализирует весь ход работы и генерирует инсайты — уроки и рекомендации для будущих проектов.

#### Что анализирует рефлексолог

- Логи выполнения всех задач
- Результаты QA-проверок
- Количество попыток и повторных выполнений
- Характер ошибок и их закономерности
- Успешность различных подходов

#### Формат инсайтов

Каждый инсайт содержит:

- **Заголовок**: Краткое название проблемы или рекомендации
- **Резюме**: Описание ситуации и что произошло
- **Категория**: TOOLING, WORKFLOW, QA_PROCESS, ARCHITECTURE, DOCUMENTATION или MISC
- **Уровень серьёзности**: low, medium или high
- **Рекомендация**: Конкретное действие для улучшения
- **Теги**: Ключевые слова для поиска

## 9. Система биллинга и метрик

Система отслеживает все затраты на AI и предоставляет детальную аналитику по проектам и всей системе.

### Трекинг затрат

Каждый вызов AI логируется с информацией:

- Количество токенов (input и output)
- Модель, которая использовалась
- Провайдер (OpenAI, Anthropic, Z.ai, OpenRouter)
- Стоимость в долларах
- Время выполнения

### Дашборд проекта

Для каждого проекта доступна информация:

- Общая сумма затрат
- Разбивка по задачам и сессиям
- Сравнение стоимости разных планов
- График затрат по времени

### Глобальные метрики

Системные метрики включают:

- Общее количество проектов и активных за последние 7 дней
- Статистика по задачам (всего, завершено, по статусам)
- Общее количество инсайтов и популярные технологии
- Совокупные затраты и количество API-вызовов

### Статистика по движкам

Отдельная аналитика для Legacy и AHP движков:

- Количество сессий
- Средняя продолжительность
- Средняя стоимость
- Процент успешных завершений
- Среднее количество шагов на сессию
- Среднее количество ошибок

## 10. RAG и контекст проекта

Система использует Retrieval-Augmented Generation для управления контекстом проекта. Это позволяет агентам иметь доступ к релевантной информации из файлов проекта при принятии решений.

### Синхронизация файлов

При изменении файлов в проекте они автоматически синхронизируются с системой:

- Файлы разбиваются на чанки
- Для каждого чанка генерируется эмбеддинг
- Чанки сохраняются в векторной базе данных (pgvector)

### Векторный поиск

Агенты могут искать по файлам проекта с помощью семантического поиска:

-natural language запросы к коду
- Поиск по названиям функций, классов
- Поиск по документации

### Извлечение сущностей кода

Система автоматически анализирует загруженные файлы и извлекает:

- Классы и их сигнатуры
- Функции и методы
- Интерфейсы и типы
- Импорты и экспорты
- Зависимости между сущностями

### Глобальный контекст

Помимо контекста отдельного проекта, система поддерживает глобальный контекст:

- Правила оформления кода (coding standards)
- Общие архитектурные принципы
- Информация о технологиях
- Предыдущие инсайты из завершённых проектов

## 11. Глобальные инсайты

Инсайты — это уроки, извлечённые из завершённых проектов и сессий выполнения. Они сохраняются глобально и используются при планировании новых проектов.

### Источники инсайтов

- Результаты работы рефлексолога
- Анализ удалённых проектов
- Ручной ввод пользователей

### Структура инсайта

Каждый инсайт содержит:

- Заголовок и резюме
- Категорию (TOOLING, WORKFLOW, QA_PROCESS, ARCHITECTURE, DOCUMENTATION, MISC)
- Уровень серьёзности
- Рекомендацию к действию
- Теги для поиска
- Привязку к проекту, плану или сессии

### Поиск инсайтов

Инсайты доступны через поиск:

- Полнотекстовый поиск по содержимому
- Фильтрация по тегам
- Фильтрация по категориям
- Семантический поиск через RAG

### Применение

При генерации новых планов AI автоматически учитывает релевантные инсайты, чтобы избежать повторения прошлых ошибок.

## 12. Интеграция с GitHub

Система интегрируется с GitHub через вебхуки для автоматического отслеживания прогресса.

### Обрабатываемые события

- **Push events**: При пуше в ветку задача переводится в статус IN_PROGRESS
- **Pull Request opened**: При открытии PR задача переводится в REVIEW
- **Pull Request merged**: При мерже PR задача переводится в DONE

### Проверка подписи

Все вебхуки верифицируются по HMAC-SHA256 подписи для защиты от поддельных запросов.

### Настройка

Для активации интеграции необходимо:

- Настроить webhook в репозитории GitHub
- Указать URL `/api/webhooks/github`
- Настроить secret в переменных окружения

## 13. MCP — Model Context Protocol

Полноценная реализация MCP-сервера, позволяющая внешним клиентам (Claude Desktop, Cursor и другим MCP-совместимым приложениям) взаимодействовать с AI Orchestrator.

### инструменты

- Доступные **get_my_tasks**: Получить список задач проекта
- **read_task**: Получить детальную информацию о задаче
- **update_task_status**: Обновить статус задачи

### Протокол

- **SSE endpoint**: `GET /api/mcp/sse` — обнаружение инструментов
- **Messages endpoint**: `POST /api/mcp/messages` — выполнение инструментов

### Применение

MCP позволяет интегрировать AI Orchestrator в сторонние IDE и AI-ассистенты, предоставляя им доступ к задачам проекта.

## 14. Система тикетов

Тикеты представляют собой отдельную систему отслеживания проблем, которая может быть связана с задачами или существовать независимо.

### Статусы тикетов

- **OPEN**: Тикет создан и ожидает обработки
- **IN_PROGRESS**: Тикет взят в работу
- **DONE**: Тикет решён
- **REJECTED**: Тикет отклонён (не воспроизводится, не требует действия и т.д.)

### Связь с задачами

Тикет может быть связан с конкретной задачей через поле `relatedTaskId`. Это позволяет:

- Отслеживать баги до конкретной реализации
- Связывать фича-реквесты с задачами реализации
- Видеть контекст тикета при работе над задачей

### Обработка при выполнении

При автоматическом выполнении открытые тикеты обрабатываются перед задачами, что позволяет сначала закрыть существующие проблемы.

## 16. Анализ кода

Система автоматически анализирует загруженные файлы проекта для извлечения структурной информации.

### Извлекаемые сущности

- **CLASS**: Классы и их сигнатуры
- **FUNCTION**: Функции и методы
- **VARIABLE**: Переменные и константы
- **IMPORT**: Операторы импорта
- **EXPORT**: Операторы экспорта
- **INTERFACE**: Интерфейсы и типы

### Типы зависимостей

- **IMPORTS**: Один файл импортирует другой
- **CALLS**: Функция вызывает другую функцию
- **EXTENDS**: Наследование классов
- **IMPLEMENTS**: Реализация интерфейсов
- **TYPE_OF**: Связь типов

### Применение

Информация о коде используется для:

- Семантического поиска по коду
- Анализа зависимостей
- Понимания структуры проекта агентами
- Автоматического обновления документации

## 18. Синхронизация файлов

Система поддерживает двустороннюю синхронизацию файлов между сервером и клиентом.

### Режимы синхронизации

- **Host sync-client (рекомендуемый)**: Синхронизатор запускается на машине разработчика
- **Server sync-client**: Синхронизатор запускается на сервере в Docker-контейнере

### Функции

- **Создание команд**: Сервер может отправлять команды на выполнение клиенту
- **Heartbeat**: Механизм keep-alive для отслеживания соединения
- **Real-time sync**: Мгновенная синхронизация изменений файлов

### Применение

Синхронизация используется для:

- Передачи файлов проекта на сервер
- Выполнения команд на удалённой машине
- Обновления локальных файлов результатами работы AI
